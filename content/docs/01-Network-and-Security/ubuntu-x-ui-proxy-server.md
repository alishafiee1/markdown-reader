# راهنمای ساده راه‌اندازی Gateway امن با اوبونتو

## 📖 مقدمه: این سند برای چه کسانی است؟

این راهنما برای کسانی نوشته شده که می‌خواهند:
- از فیلترینگ اینترنت عبور کنند
- یک سرور شخصی برای خود و دوستانشان راه‌اندازی کنند
- بدانند که فناوری‌های امنیتی چگونه کار می‌کنند
- از دانش فنی متوسط برخوردار هستند اما به توضیح ساده نیاز دارند

---

## 🎯 هدف نهایی: چه چیزی می‌خواهیم بسازیم؟

یک سرور میانی (Gateway) که:
1. **ترافیک اینترنت شما را مسیردهی می‌کند** از یک اینترنت آزاد
2. **در برابر سیستم‌های کنترل و نظارت مقاوم است** (مثل DPI)
3. **به نظر یک ترافیک عادی می‌رسد** نه یک پروکسی یا VPN
4. **برای چندین کاربر قابل استفاده است** (خانواده، دوستان)

---

## 🧩 قسمت اول: درک مفاهیم پایه

### 1.1 Gateway چیست؟

**تعریف ساده:** Gateway مثل یک "دروازه‌بان" است که تمام ترافیک شبکه شما را از خودش عبور می‌دهد و به اینترنت می‌فرستد.

**مثال واقعی:** 
```
[کامپیوتر شما] ---> [Gateway = سرور اوبونتو] ---> [اینترنت آزاد]
```

**چرا نیاز داریم؟**
- مودم اصلی شما فیلتر دارد
- Gateway از اینترنت دیگری (بدون فیلتر) استفاده می‌کند
- شما به جای مودم فیلترشده، از Gateway استفاده می‌کنید

---

### 1.2 پروکسی چیست و چه تفاوتی با VPN دارد؟

**پروکسی:**
- یک سرور واسط که درخواست‌های شما را می‌گیرد و از طرف شما به اینترنت می‌فرستد
- فقط برنامه‌های خاصی (مثل مرورگر) می‌توانند از آن استفاده کنند
- سریع‌تر از VPN است

**VPN:**
- تمام ترافیک سیستم شما را رمزنگاری می‌کند
- همه برنامه‌ها (حتی بازی‌ها و نرم‌افزارها) از آن استفاده می‌کنند
- کمی کندتر است اما امن‌تر

**در این پروژه ما از روش ترکیبی استفاده می‌کنیم:** پروکسی با قابلیت‌های امنیتی بالا

---

### 1.3 DPI چیست و چرا مشکل‌ساز است؟

**DPI = Deep Packet Inspection = بازرسی عمیق بسته‌ها**

**تعریف ساده:**
یک فناوری که دولت‌ها و اپراتورها استفاده می‌کنند تا محتوای داخل بسته‌های اینترنتی شما را بررسی کنند (نه فقط مقصد).

**مثال:**
- بدون DPI: سیستم فقط می‌بیند "شما به آدرس X وصل شدید"
- با DPI: سیستم می‌بیند "شما به آدرس X وصل شدید + دارید چه محتوایی رد و بدل می‌کنید + از چه نرم‌افزاری استفاده می‌کنید"

**چطور کار می‌کند؟**

1. **بررسی SNI:**
   - SNI = Server Name Indication
   - وقتی به یک سایت HTTPS وصل می‌شوید، در ابتدای اتصال نام سایت را به صورت رمزنگاری نشده می‌فرستید
   - DPI این نام را می‌خواند و اگر فیلتر باشد، قطع می‌کند

2. **مقایسه اثرانگشت (Fingerprinting):**
   - هر نرم‌افزار (Chrome, Firefox, VPN, پروکسی) یک الگوی خاص در ارتباط TLS دارد
   - DPI این الگوها را می‌شناسد و می‌گوید "این یک پروکسی است!"

3. **بازرسی فعال (Active Probing):**
   - سیستم DPI خودش یک درخواست تست به سرور شما می‌فرستد
   - اگر پاسخ مشکوک باشد، IP شما را مسدود می‌کند

4. **تحلیل رفتاری:**
   - الگوی استفاده شما را بررسی می‌کند (حجم، زمان، نوع داده)
   - اگر غیرعادی باشد (مثل ترافیک سنگین مداوم)، مشکوک می‌شوید

---

### 1.4 تکنولوژی REALITY چیست؟

**تعریف:**
REALITY یک روش جدید برای عبور از فیلترینگ است که در سال 2023 معرفی شد.

**چگونه کار می‌کند؟**

به جای اینکه "تقلب کند" و خودش را جای سایت دیگری جا بزند، REALITY **از یک سایت واقعی به عنوان آینه استفاده می‌کند**.

**مثال عملی:**

```
کاربر ---> درخواست ---> سرور شما ---> بررسی اعتبار ---> OK/NOT OK
                                        |
                                        +---> اگر OK: به پروکسی وصل شو
                                        |
                                        +---> اگر NOT OK: به microsoft.com وصل شو
```

**چرا DPI نمی‌تواند تشخیص دهد؟**
- درخواست شما دقیقاً مثل یک درخواست واقعی به Microsoft است
- اثرانگشت TLS دقیقاً مثل Chrome است
- حتی اگر DPI بازرسی کند، یک سایت واقعی می‌بیند

## 🏗️ قسمت دوم: معماری سیستم

### 2.1 نقشه کلی ساختار

```
                          ┌──────────────────────────┐
                          │   کاربران (شما/دوستان)  │
                          └────────────┬─────────────┘
                                      │
                           درخواست پروکسی (VLESS+REALITY)
                                      │
                                      ▼
                          ┌───────────────────────────┐
                          │   سرور اوبونتو Gateway    │
                          │                           │
                          │  ┌─────────────────────┐  │
                          │  │    پنل X-UI         │  │
                          │  │  + Xray-Core        │  │
                          │  └─────────────────────┘  │
                          │                           │
                          │  [eth0]        [eth1]     │
                          └────┬──────────────┬───────┘
                               │              │
                    به اینترنت آزاد     به مودم فیلترشده
                      (خروجی اصلی)      (فقط برای مدیریت)
```

### 2.2 نقش هر قسمت

**سرور اوبونتو:**
- سیستم عامل لینوکس که همیشه روشن است
- دو کارت شبکه دارد (دو پورت LAN)

**پنل X-UI:**
- یک رابط کاربری وب برای مدیریت پروکسی
- از طریق مرورگر قابل دسترسی است
- می‌توانید کاربر اضافه کنید، ترافیک ببینید، تنظیمات تغییر دهید

**Xray-Core:**
- موتور اصلی پروکسی که پشت X-UI کار می‌کند
- پروتکل VLESS و REALITY را پشتیبانی می‌کند

**eth0 (پورت اول):**
- به شبکه با اینترنت آزاد وصل است
- تمام ترافیک کاربران از اینجا خارج می‌شود

**eth1 (پورت دوم):**
- به مودم سیم‌کارتی وصل است
- فقط برای مدیریت سرور استفاده می‌شود

## 🔧 قسمت سوم: نصب و راه‌اندازی (گام به گام)

### مرحله 1: آماده‌سازی سرور اوبونتو

**هدف:** سیستم را برای مسیردهی ترافیک آماده کنیم.

```bash
# به روزرسانی سیستم
sudo apt update && sudo apt upgrade -y

# نصب ابزارهای ضروری
sudo apt install -y curl wget ufw iptables-persistent

# فعال کردن IP Forwarding (مسیردهی ترافیک بین دو شبکه)
echo 'net.ipv4.ip_forward=1' | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
```

**توضیح IP Forwarding:**
- به طور پیش‌فرض، لینوکس بسته‌های شبکه را از یک کارت به کارت دیگر منتقل نمی‌کند
- با فعال کردن این گزینه، سرور مثل یک Router عمل می‌کند

---

### مرحله 2: تنظیم NAT

**هدف:** ترافیکی که از کاربران می‌آید را از طریق اینترنت آزاد خارج کنیم.

```bash
# فعال کردن NAT برای eth0 (پورتی که به اینترنت آزاد وصل است)
sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# ذخیره قوانین
sudo netfilter-persistent save
```

**توضیح NAT (Network Address Translation):**
- وقتی کاربر از شما درخواست می‌دهد، IP او محلی است (مثل 192.168.1.5)
- NAT این IP را به IP سرور شما تبدیل می‌کند
- سایت مقصد فکر می‌کند درخواست از سرور شما آمده، نه کاربر

**مثال:**
```
کاربر (192.168.1.5) ---> Gateway --[NAT]---> (IP عمومی Gateway) ---> Google
```

### مرحله 4: دریافت دامنه و راه‌اندازی Fallback

**چرا به دامنه نیاز داریم؟**
- برای داشتن یک هویت قانونی در اینترنت
- DPI وقتی IP شما را بررسی کند، یک سایت واقعی ببیند

**مراحل:**

**4.1 خرید دامنه:**
- از سایت‌هایی مثل Namecheap, Cloudflare, یا سایت‌های ایرانی
- یک دامنه ارزان (مثل `.xyz`, `.site`) کافی است - حدود $1-3 سالانه

**4.2 تنظیم DNS:**
```
نوع: A Record
نام: @ (یا example.com)
مقدار: IP سرور شما
TTL: 300 یا Auto
```

**4.3 نصب Nginx (وب‌سرور ساده):**

```bash
sudo apt install -y nginx

# ایجاد یک صفحه ساده
sudo nano /var/www/html/index.html
```

**محتوای نمونه برای index.html:**

```html
<!DOCTYPE html>
<html lang="fa" dir="rtl">
<head>
    <meta charset="UTF-8">
    <title>صفحه شخصی من</title>
    <style>
        body { font-family: Tahoma; text-align: center; padding: 50px; }
    </style>
</head>
<body>
    <h1>به سایت من خوش آمدید</h1>
    <p>این یک صفحه شخصی ساده است.</p>
</body>
</html>
```

**چرا این کار را انجام می‌دهیم?**
- وقتی DPI به سرور شما وصل شود (بازرسی فعال)، یک سایت واقعی ببیند
- این به "Fallback" معروف است - یعنی اگر کسی بدون رمز صحیح بیاید، به این صفحه هدایت شود

### مرحله 5: ایجاد Inbound در X-UI با REALITY

**5.1 ورود به پنل X-UI**

**5.2 رفتن به بخش "Inbounds"**

**5.3 کلیک روی "Add Inbound"**

**5.4 تنظیمات:**

```
───────────────────────────────────────
📋 پارامتر               │ مقدار
───────────────────────────────────────
Protocol                  │ VLESS
Port                      │ 443
Network                   │ TCP
Security                  │ REALITY
───────────────────────────────────────
REALITY Settings:
───────────────────────────────────────
  Enable REALITY          │ ✓ (تیک بزنید)
  Server Name (SNI)       │ www.microsoft.com
  Destination (dest)      │ www.microsoft.com:443
  Short ID                │ abc123 (دلخواه، 4-8 کاراکتر)
  Private Key             │ (خودکار تولید می‌شود)
  Public Key              │ (خودکار تولید می‌شود)
───────────────────────────────────────
Flow                      │ xtls-rprx-vision
───────────────────────────────────────
Fallback                  │ 127.0.0.1:80
───────────────────────────────────────
Client Settings:
───────────────────────────────────────
  UUID                    │ (خودکار تولید می‌شود)
  Email/Name              │ user1
  Traffic Limit           │ 100 GB (مثال)
───────────────────────────────────────
```

**توضیح هر پارامتر:**

**Protocol (پروتکل):**
- VLESS: جدیدترین پروتکل پروکسی - سبک و سریع
- جایگزین VMess (قدیمی‌تر)

**Port (پورت):**
- 443: پورت استاندارد HTTPS
- استفاده از این پورت باعث می‌شود ترافیک شما عادی به نظر برسد

**Security:**
- REALITY: همان فناوری ای که توضیح دادیم

**Server Name (SNI):**
- دامنه‌ای که می‌خواهید تقلید کنید
- بهترین انتخاب: سایت‌های بزرگ و پرسرعت
  - `www.microsoft.com`
  - `www.apple.com`
  - `www.cloudflare.com`

**Destination:**
- همان دامنه‌ای که در SNI گذاشتید
- REALITY ترافیک غیرمجاز را به این آدرس می‌فرستد

**Short ID:**
- یک کد شناسایی کوتاه برای کلاینت شما
- می‌تواند هر چیزی باشد (مثل `a1b2`, `myid`, `test123`)

**Flow:**
- `xtls-rprx-vision`: یک تکنولوژی برای شبیه‌سازی ترافیک ویدئو
- باعث می‌شود الگوی ترافیک شما غیرقابل تشخیص باشد

**Fallback:**
- آدرس وب‌سرور محلی شما (Nginx)
- وقتی کسی بدون رمز صحیح بیاید، به اینجا هدایت می‌شود

### مرحله 8: تنظیم فایروال

**هدف:** فقط پورت‌های ضروری را باز کنیم.

```bash
# غیرفعال کردن فایروال موقتاً (اگر فعال است)
sudo ufw disable

# تنظیمات جدید
sudo ufw default deny incoming
sudo ufw default allow outgoing

# باز کردن پورت‌های ضروری
sudo ufw allow 22/tcp     # SSH
sudo ufw allow 443/tcp    # پروکسی و Fallback

# فعال کردن فایروال
sudo ufw enable
```

**نکته امنیتی:** پورت SSH را تغییر دهید!

```bash
sudo nano /etc/ssh/sshd_config

# خط زیر را پیدا کنید:
# Port 22

# به این تغییر دهید:
Port 46522

# ذخیره و خروج (Ctrl+O, Enter, Ctrl+X)

# راه‌اندازی مجدد SSH
sudo systemctl restart sshd

# باز کردن پورت جدید
sudo ufw allow 46522/tcp
sudo ufw delete allow 22/tcp
```

## 🛡️ قسمت چهارم: امنیت پیشرفته

### 4.1 مخفی کردن پنل X-UI

**مشکل:**
- پنل X-UI اگر در معرض اینترنت باشد، هدف حملات می‌شود

**راه حل: SSH Tunneling**

```bash
# از کامپیوتر خودتان (نه سرور) این دستور را اجرا کنید:
ssh -L 8080:localhost:54321 username@IP-سرور -p 46522
```

**معنی دستور:**
- پورت 54321 سرور را روی پورت 8080 کامپیوتر خودتان باز کنید
- حالا با رفتن به `http://localhost:8080` به پنل دسترسی دارید
- از اینترنت قابل دسترسی نیست

**بهبود:**
```bash
# بستن پورت پنل در فایروال
sudo ufw delete allow 54321/tcp
```

---

### 4.2 کلید SSH به جای رمز عبور

**مزایا:**
- حمله Brute Force غیرممکن می‌شود
- امن‌تر از رمز عبور

**مراحل:**

**در کامپیوتر خودتان (نه سرور):**

```bash
# تولید کلید
ssh-keygen -t ed25519 -C "my-server-key"

# کپی کلید به سرور
ssh-copy-id -p 46522 username@IP-سرور
```

**در سرور:**

```bash
# ویرایش تنظیمات SSH
sudo nano /etc/ssh/sshd_config

# این خطوط را تغییر دهید:
PasswordAuthentication no
PubkeyAuthentication yes
PermitRootLogin no

# ذخیره و راه‌اندازی مجدد
sudo systemctl restart sshd
```

- اگر یک کاربر بیش از 500 GB مصرف کرد، مشکوک است

### 4.4 تغییر دوره‌ای تنظیمات

**برای جلوگیری از تشخیص الگو:**

**هر ماه یکبار:**
- دامنه `dest` در REALITY را تغییر دهید
- مثال: ماه اول `microsoft.com`، ماه دوم `apple.com`

**هر سه ماه:**
- `Short ID` را تغییر دهید
- لینک کانفیگ جدید به کاربران بدهید

## 🧠 قسمت پنجم: درک عمیق چالش‌ها

### 5.1 جدول مقایسه: DPI در برابر ضد DPI

| تکنیک DPI | چگونه کار می‌کند | راه حل ضد DPI ما | چرا جواب می‌دهد |
|-----------|------------------|------------------|------------------|
| **بررسی SNI** | نام سایت را در ابتدای اتصال می‌خواند | استفاده از دامنه شخصی + REALITY | SNI ما به سایت واقعی اشاره می‌کند، نه پروکسی |
| **اثرانگشت TLS** | پارامترهای اتصال را با پایگاه داده مقایسه می‌کند | `fingerprint: chrome` در REALITY | رفتار ما دقیقاً مثل Chrome است |
| **بازرسی فعال** | درخواست تست به سرور می‌فرستد | Fallback به Nginx | DPI یک سایت HTML عادی می‌بیند |
| **تحلیل رفتاری** | الگوی حجم و زمان ترافیک را بررسی می‌کند | `xtls-rprx-vision` flow | ترافیک شبیه ویدئو استریمینگ است |
| **تطابق SNI/IP** | چک می‌کند که آیا SNI با IP مقصد مطابقت دارد | دامنه شخصی به IP ما point می‌کند | تطابق کامل وجود دارد |

---

### 5.2 چرخه زندگی یک درخواست (Step by Step)

**حالت 1: کاربر مجاز**

```
1. کاربر لینک کانفیگ دارد (با UUID و Short ID صحیح)
   ↓
2. برنامه کلاینت (Nekobox) درخواست به سرور می‌فرستد
   ↓
3. سرور REALITY بررسی می‌کند: UUID و Short ID درست است؟
   ↓
4. ✓ بله → کاربر به Xray-Core وصل می‌شود
   ↓
5. ترافیک کاربر از طریق eth0 به اینترنت آزاد می‌رود
   ↓
6. پاسخ برمی‌گردد به کاربر
```

**حالت 2: کاربر غیرمجاز (مثل DPI)**

```
1. یک سیستم ناشناس (مثلاً DPI) به IP شما وصل می‌شود
   ↓
2. اطلاعات صحیح (UUID, Short ID) ندارد
   ↓
3. سرور REALITY بررسی می‌کند: اطلاعات نادرست
   ↓
4. ✗ خیر → به Fallback هدایت می‌شود (Nginx)
   ↓
5. DPI یک صفحه HTML معمولی می‌بیند
   ↓
6. نتیجه: "این یک سایت عادی است، نه پروکسی"
```

---

### 5.3 مقایسه با روش‌های قدیمی

| روش | مشکل اصلی | چرا قدیمی است |
|-----|----------|----------------|
| **Shadowsocks** | اثرانگشت TLS خاص | DPI به راحتی تشخیص می‌دهد |
| **VMess** | رمزنگاری سنگین و الگوی قابل تشخیص | برای هر بسته overhead دارد |
| **Trojan** | نیاز به دامنه معتبر با گواهی SSL | اگر گواهی منقضی شود، قطع می‌شود |
| **OpenVPN** | هدرهای خاص و قابل فیلتر | یکی از اولین چیزهایی که DPI می‌شناسد |
| **WireGuard** | پورت UDP و الگوی ثابت | به راحتی با بررسی UDP شناسایی می‌شود |

**چرا REALITY بهتر است؟**
- هیچ الگوی ثابتی ندارد
- از سایت‌های واقعی به عنوان "پوشش" استفاده می‌کند
- اثرانگشت TLS آن دقیقاً مثل مرورگرهای معمولی است
- در برابر بازرسی فعال مقاوم است

### 7.2 مفهوم Multiplexing (Mux)

**تعریف ساده:**
- به جای باز کردن یک اتصال برای هر درخواست، چندین درخواست را در یک اتصال جا می‌دهیم

**مزایا:**
- کاهش overhead
- سریع‌تر برای وب‌سایت‌هایی با تصاویر زیاد

**فعال‌سازی در کلاینت:**
```
Nekobox > Edit Profile > Advanced > Mux: Enable
Concurrent Connections: 8
```

### 7.3 DNS و تأثیر آن

**مشکل:** حتی اگر ترافیک شما رمز باشد، DNS شما را لو می‌دهد!

**راه حل:**

**1. استفاده از DNS over HTTPS (DoH):**

در کلاینت (مثلاً Nekobox):
```
Settings > DNS > DoH > https://1.1.1.1/dns-query
```

**2. تنظیم DNS در سرور:**

```bash
sudo nano /etc/systemd/resolved.conf

# تغییر به:
[Resolve]
DNS=1.1.1.1 8.8.8.8
FallbackDNS=9.9.9.9
DNSOverTLS=yes

# راه‌اندازی مجدد
sudo systemctl restart systemd-resolved
```

## 🔐 قسمت هشتم: امنیت لایه‌ای (Defense in Depth)

### 8.1 اصل لایه‌های امنیتی

**ایده اصلی:** اگر یک لایه شکست خورد، لایه‌های دیگر محافظت می‌کنند.

```
لایه 1: Firewall (فقط پورت 443 باز)
   ↓
لایه 2: REALITY (بررسی UUID و Short ID)
   ↓
لایه 3: Fallback (DPI یک سایت عادی می‌بیند)
   ↓
لایه 4: SSH با کلید (نه رمز عبور)
   ↓
لایه 5: تغییر دوره‌ای تنظیمات
```

---

### 8.2 قوانین طلایی امنیت

1. **Principle of Least Privilege** (کمترین دسترسی ممکن)
   - فقط پورت‌های ضروری باز باشند
   - هر کاربر فقط به ترافیک خودش دسترسی داشته باشد

2. **Defense in Depth** (امنیت لایه‌ای)
   - یک لایه امنیتی کافی نیست

3. **Security through Obscurity** (مخفی کردن)
   - تغییر پورت SSH
   - تغییر مسیر پنل X-UI
   - عدم اشتراک IP سرور در عموم

4. **Regular Updates** (به‌روزرسانی منظم)
   - هر ماه سیستم را آپدیت کنید
   - Xray-Core را به روز نگه دارید

## 📚 واژه‌نامه (Glossary)

| اصطلاح | معنی فارسی | توضیح ساده |
|--------|------------|-----------|
| **Gateway** | دروازه/درگاه | سیستمی که ترافیک شبکه را مسیردهی می‌کند |
| **Proxy** | پروکسی/واسط | سرور میانی که درخواست‌های شما را جابجا می‌کند |
| **VPN** | شبکه خصوصی مجازی | تونلی رمزشده برای تمام ترافیک سیستم |
| **DPI** | بازرسی عمیق بسته‌ها | فناوری بررسی محتوای داخل بسته‌های شبکه |
| **SNI** | نشانگر نام سرور | نام سایتی که در ابتدای اتصال HTTPS ارسال می‌شود |
| **TLS** | امنیت لایه انتقال | پروتکل رمزنگاری برای HTTPS |
| **Fingerprint** | اثرانگشت | الگوی منحصربفرد یک نرم‌افزار در اتصال شبکه |
| **Fallback** | بازگشت/جایگزین | جایی که ترافیک غیرمجاز هدایت می‌شود |
| **NAT** | ترجمه آدرس شبکه | تبدیل IP خصوصی به عمومی |
| **Forwarding** | مسیردهی | انتقال بسته از یک شبکه به شبکه دیگر |
| **Inbound** | ورودی | تنظیمات پذیرش کاربران در سرور پروکسی |
| **UUID** | شناسه یکتای جهانی | یک کد منحصربفرد برای هر کاربر |
| **Flow** | جریان | روش مدیریت ترافیک (مثل vision) |
| **Mux** | چندگانه‌سازی | ارسال چند درخواست در یک اتصال |
| **DoH** | DNS روی HTTPS | رمزنگاری درخواست‌های DNS |
