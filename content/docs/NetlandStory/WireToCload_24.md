### فصل بیست و چهارم: ابزارهای عیبیابی (با دست خالی)

---

#### بخش اول: مقدمه

در دنیای شبکه، گاهی بدون هیچ ابزار خاصی و تنها با استفاده از امکانات خط فرمان می‌توان مشکلات را شناسایی و رفع کرد. در این فصل با ابزارهای ساده اما قدرتمندی آشنا می‌شویم که در ویندوز، مک، لینوکس و MikroTik در دسترس هستند.

---

#### بخش دوم: Ping - بررسی زنده بودن دستگاه

**Ping** ساده‌ترین و پرکاربردترین ابزار عیب‌یابی است. این ابزار با ارسال بسته‌های ICMP به مقصد، زنده بودن دستگاه و میزان تأخیر را بررسی می‌کند.

**نحوه‌ی کار:**
1. یک بسته‌ی درخواست پژواک (Echo Request) به مقصد می‌فرستد
2. اگر مقصد زنده باشد، بسته‌ی پاسخ پژواک (Echo Reply) برمی‌گرداند
3. زمان رفت و برگشت (RTT) محاسبه می‌شود

**کاربردها:**
- بررسی اتصال به اینترنت
- تست زنده بودن یک سرور یا دستگاه
- اندازه‌گیری تأخیر (Latency)
- تشخیص قطعی (اگر پاسخ نیاید)

**دستورات:**

| سیستم‌عامل | دستور | مثال |
|-----------|-------|-------|
| ویندوز | `ping [آدرس]` | `ping google.com` |
| مک/لینوکس | `ping [آدرس]` | `ping 8.8.8.8` |
| MikroTik | `/ping [آدرس]` | `/ping 8.8.8.8` |

**گزینه‌های مفید:**

| گزینه | توضیح | مثال |
|-------|-------|-------|
| `-t` (ویندوز) | پینگ مداوم تا دستور قطع | `ping -t google.com` |
| `-n [تعداد]` (ویندوز) | تعداد بسته‌ها | `ping -n 100 google.com` |
| `-c [تعداد]` (مک/لینوکس) | تعداد بسته‌ها | `ping -c 100 google.com` |
| `-l [اندازه]` (ویندوز) | اندازه‌ی بسته | `ping -l 1400 google.com` |
| `-s [اندازه]` (مک/لینوکس) | اندازه‌ی بسته | `ping -s 1400 google.com` |

**تفسیر نتایج:**

| نتیجه | معنی |
|-------|-------|
| `Reply from ...` | دستگاه زنده است |
| `Request timed out` | پاسخی نیامد (قطع یا فایروال) |
| `Destination unreachable` | مسیر به مقصد وجود ندارد |
| `TTL=...` | زمان زندگی بسته (تعداد گام‌های باقی‌مانده) |
| `time=...ms` | زمان رفت و برگشت (میلی‌ثانیه) |

---

#### بخش سوم: Traceroute - مسیر رسیدن به مقصد

**Traceroute** مسیر بسته‌ها را از مبدأ تا مقصد نشان می‌دهد. این ابزار برای تشخیص محل قطعی یا تأخیر در مسیر بسیار مفید است.

**نحوه‌ی کار:**
1. بسته‌هایی با TTL (Time to Live) افزایشی ارسال می‌کند
2. هر روتر در مسیر که TTL تمام شود، پیام خطا برمی‌گرداند
3. با جمع‌آوری این پاسخ‌ها، مسیر کامل مشخص می‌شود

**دستورات:**

| سیستم‌عامل | دستور | مثال |
|-----------|-------|-------|
| ویندوز | `tracert [آدرس]` | `tracert google.com` |
| مک/لینوکس | `traceroute [آدرس]` | `traceroute google.com` |
| MikroTik | `/tool traceroute [آدرس]` | `/tool traceroute google.com` |

**تفسیر نتایج:**

| ستون | توضیح |
|------|-------|
| شماره‌ی گام (Hop) | شماره‌ی روتر در مسیر |
| آدرس IP | آدرس روتر |
| زمان (ms) | زمان رسیدن به آن روتر (معمولاً ۳ بار) |
| `* * *` | پاسخ نیامد (روتر پاسخ نمی‌دهد یا مسیر قطع است) |

**کاربردها:**
- تشخیص محل قطعی (ببینید بسته در کدام گام متوقف می‌شود)
- تشخیص تأخیر (ببینید کدام گام تأخیر ایجاد می‌کند)
- مشاهده‌ی مسیرهای مختلف

---

#### بخش چهارم: Netstat - بررسی اتصالات فعال

**Netstat** (Network Statistics) اطلاعاتی درباره‌ی اتصالات شبکه، پورت‌های باز و آمار شبکه نمایش می‌دهد.

**دستورات پایه:**

| دستور | توضیح |
|-------|-------|
| `netstat -a` | نمایش همه‌ی اتصالات و پورت‌های باز |
| `netstat -n` | نمایش آدرس‌ها به صورت عددی (بدون حل نام) |
| `netstat -an` | ترکیب دو گزینه‌ی بالا |
| `netstat -b` (ویندوز) | نمایش برنامه‌ی هر اتصال (نیاز به اجرا به عنوان admin) |
| `netstat -r` | نمایش جدول مسیریابی |

**نمونه خروجی در ویندوز:**
```
Proto  Local Address          Foreign Address        State
TCP    192.168.1.10:54321     172.217.168.46:443     ESTABLISHED
TCP    192.168.1.10:54322     13.107.42.14:80        TIME_WAIT
UDP    192.168.1.10:53        *:*                   
```

**تفسیر ستون‌ها:**

| ستون | توضیح |
|-------|-------|
| `Proto` | پروتکل (TCP یا UDP) |
| `Local Address` | آدرس و پورت دستگاه شما |
| `Foreign Address` | آدرس و پورت دستگاه مقصد |
| `State` | وضعیت اتصال (فقط TCP) |

**وضعیت‌های مهم TCP:**

| وضعیت | توضیح |
|-------|-------|
| `ESTABLISHED` | اتصال فعال برقرار است |
| `LISTENING` | برنامه منتظر اتصال است (پورت باز) |
| `TIME_WAIT` | اتصال بسته شده، در حال انتظار |
| `CLOSE_WAIT` | طرف مقابل اتصال را بسته، برنامه هنوز نبسته |
| `SYN_SENT` | درخواست اتصال فرستاده شده، منتظر پاسخ |

**کاربردها:**
- شناسایی اتصالات مشکوک
- بررسی اینکه کدام برنامه از شبکه استفاده می‌کند
- عیب‌یابی مشکلات اتصال

---

#### بخش پنجم: Nslookup - پرس و جوی DNS

**Nslookup** (Name Server Lookup) برای پرس و جو از سرورهای DNS و بررسی عملکرد دفترچه‌ی تلفن اینترنت استفاده می‌شود.

**دستورات پایه:**

| دستور | توضیح |
|-------|-------|
| `nslookup [دامنه]` | نمایش آدرس IP دامنه |
| `nslookup [IP]` | نمایش نام دامنه (reverse lookup) |
| `nslookup` | وارد حالت تعاملی می‌شود |

**نمونه خروجی:**
```
> nslookup google.com
Server:  dns.google
Address:  8.8.8.8

Non-authoritative answer:
Name:    google.com
Addresses:  2a00:1450:4001:830::200e
          172.217.168.46
```

**کاربردها:**
- بررسی اینکه آیا DNS درست کار می‌کند
- یافتن آدرس IP یک سایت
- بررسی رکوردهای مختلف DNS (MX، TXT، CNAME)

**بررسی انواع رکوردهای DNS:**

| رکورد | کاربرد | دستور |
|-------|-------|-------|
| **A** | IPv4 | `nslookup -type=A google.com` |
| **AAAA** | IPv6 | `nslookup -type=AAAA google.com` |
| **MX** | سرور ایمیل | `nslookup -type=MX gmail.com` |
| **NS** | سرورهای نام | `nslookup -type=NS google.com` |
| **TXT** | متون (SPF, DKIM) | `nslookup -type=TXT google.com` |

#### بخش ششم: Ipconfig / Ifconfig - اطلاعات شبکه

**Ipconfig** (ویندوز) و **Ifconfig** (مک/لینوکس) اطلاعات مربوط به پیکربندی شبکه را نمایش می‌دهند.

**دستورات ویندوز:**

| دستور | توضیح |
|-------|-------|
| `ipconfig` | نمایش اطلاعات پایه (IP، subnet mask، gateway) |
| `ipconfig /all` | نمایش اطلاعات کامل (MAC، DNS، DHCP، ...) |
| `ipconfig /release` | آزاد کردن آدرس IP (برای DHCP) |
| `ipconfig /renew` | دریافت مجدد آدرس IP از DHCP |
| `ipconfig /flushdns` | پاک کردن کش DNS |
| `ipconfig /displaydns` | نمایش کش DNS |

**دستورات مک/لینوکس:**

| دستور | توضیح |
|-------|-------|
| `ifconfig` | نمایش اطلاعات تمام اینترفیس‌ها |
| `ifconfig [interface]` | نمایش اطلاعات یک اینترفیس خاص |
| `ip addr show` | نمایش آدرس‌های IP (دستور جدیدتر) |
| `ip route show` | نمایش جدول مسیریابی |

**کاربردها:**
- پیدا کردن آدرس IP دستگاه
- مشاهده‌ی آدرس MAC
- عیب‌یابی مشکل عدم دریافت IP
- پاک کردن کش DNS برای رفع مشکلات دسترسی به سایت‌ها

---

#### بخش هفتم: Arp - مشاهده‌ی همسایه‌ها

**ARP** (Address Resolution Protocol) جدول تبدیل آدرس IP به MAC را نشان می‌دهد.

**دستورات:**

| سیستم‌عامل | دستور | توضیح |
|-----------|-------|-------|
| ویندوز | `arp -a` | نمایش جدول ARP |
| ویندوز | `arp -d` | پاک کردن جدول ARP |
| مک/لینوکس | `arp -a` | نمایش جدول ARP |
| MikroTik | `/ip arp print` | نمایش جدول ARP |

**کاربردها:**
- مشاهده‌ی دستگاه‌های متصل به شبکه
- تشخیص حملات ARP Spoofing
- بررسی اینکه آیا دستگاه‌ها در شبکه شناسایی می‌شوند

---

#### بخش هشتم: کارگاه عملی

**هدف:** استفاده از ابزارهای عیب‌یابی برای شناسایی مشکلات رایج شبکه.

**ابزار:** کامپیوتر با ویندوز (یا مک/لینوکس).

**مراحل:**

**۱. تست اتصال پایه با Ping**

- خط فرمان را باز کنید.
- دستور `ping 8.8.8.8` را بزنید.
- اگر پاسخ گرفتید، اینترنت شما کار می‌کند.
- دستور `ping google.com` را بزنید.
- اگر پاسخ گرفتید، DNS کار می‌کند. اگر نگرفتید، مشکل DNS دارید.

**۲. تشخیص محل قطعی با Traceroute**

- دستور `tracert google.com` را بزنید.
- ببینید بسته در کدام گام متوقف می‌شود.
- اگر در گام اول متوقف شد، مشکل مودم یا روتر شماست.
- اگر در گام‌های بعدی متوقف شد، مشکل از ارائه‌دهنده یا خارج از کشور است.

**۳. بررسی اتصالات فعال با Netstat**

- دستور `netstat -an` را بزنید.
- تعداد اتصالات `ESTABLISHED` را ببینید.
- اگر تعداد اتصالات غیرعادی زیاد بود، ممکن است بدافزار داشته باشید.
- برای دیدن برنامه‌ی هر اتصال (ویندوز): `netstat -abn` (اجرا به عنوان admin)

**۴. تست عملکرد DNS با Nslookup**

- دستور `nslookup google.com` را بزنید.
- آدرس IP برگشتی را یادداشت کنید.
- دستور `nslookup 8.8.8.8` را بزنید (reverse lookup).
- اگر نام دامنه برگرداند، DNS معکوس کار می‌کند.

**۵. بررسی اطلاعات شبکه با Ipconfig**

- دستور `ipconfig /all` را بزنید.
- آدرس IP، MAC، DNS و Gateway خود را پیدا کنید.
- اگر IP با `169.254.x.x` شروع می‌شود، DHCP کار نمی‌کند.
- برای رفع مشکل، دستورات زیر را به ترتیب بزنید:
  ```
  ipconfig /release
  ipconfig /renew
  ```

**۶. مشاهده‌ی همسایه‌ها با Arp**

- دستور `arp -a` را بزنید.
- آدرس Gateway (Default Gateway) را پیدا کنید.
- آدرس MAC آن را یادداشت کنید.
- اگر روزی این آدرس MAC تغییر کرد بدون اینکه مودم عوض شده باشد، احتمال حمله ARP Spoofing وجود دارد.
