## CI/CD چیست؟ (با زبون ساده و آموزشی)

خیلی ساده بگم: **CI/CD یعنی خودکار کردن مسیر رسیدن کد تو از لپ‌تاپت تا سرور زنده!**

فرض کن هر بار که یه تغییری توی کدت میدی، باید:
- کد رو تست کنی
- خطاها رو پیدا کنی
- فایل‌ها رو فشرده کنی
- به سرور منتقل کنی
- سرویس رو ری‌استارت کنی

**CI/CD می‌گه: "بیا همه این کارها رو به ربات بسپار!"**

---

## شکستن مفهوم به دو قسمت ساده

### 🟢 CI = Continuous Integration (یکپارچه‌سازی پیوسته)

**یعنی:** چندین بار در روز، هر تغییری که توی کدت میدی، بلافاصله با بقیه کدها قاطی می‌شه و تست می‌شه.

**مثال واقعی:** فرض کن ۳ تا برنامه‌نویس توی یه تیم هستین:
- من توی فایل `user.js` کار می‌کنم
- تو توی فایل `payment.js` کار می‌کنی
- علی توی `database.js` کار می‌کنه

وقتی من کدم رو به گیت‌هاب می‌فرستم (Push)، **CI** به طور خودکار:
1. کد من رو با کد شما و علی قاطی می‌کنه (Merge)
2. همه تست‌ها رو اجرا می‌کنه
3. اگر خطایی باشه، بهم ایمیل می‌زنه "فلان جا خراب کردی!"

**نتیجه:** دیگه آخر هفته بیدار نمی‌مونی تا ۵ ساعت خطای قاطی شدن کدها رو پیدا کنی!

---

### 🔵 CD = Continuous Delivery/Deployment (تحویل/استقرار پیوسته)

**یعنی:** بعد از اینکه CI گفت "کد سالم است"، خودکار کد رو به سرورهای مختلف بفرست.

**دو نوع CD داریم:**

| نوع | معنی | مثال |
|-----|------|------|
| **Continuous Delivery** | کد آماده است، ولی تو خودت دکمه بزن بره روی سرور | "آقا جان، کدت تست شد، فقط یه دکمه بزن بره روی سایت اصلی" |
| **Continuous Deployment** | کاملاً خودکار! بدون دخالت انسان | "کد ریخت روی گیت‌هاب → خودکار تست شد → خودکار رفت روی سرور زنده" |

---

## جریان کامل CI/CD (از لحظه نوشتن کد تا دیده شدن در وب)

```
۱. تو کد رو توی لپ‌تاپت می‌نویسی
     ↓
۲. دستور میدی: git push origin main
     ↓
۳. گیت‌هاب (یا گیت‌لب) می‌فهمد کد جدید اومده → شروع کن به CI/CD
     ↓
۴. مرحله CI شروع می‌شه:
   • دانلود وابستگی‌ها (npm install یا pip install)
   • اجرای تست‌ها (pytest یا npm test)
   • چک کردن کیفیت کد (لینترها: ESLint, Black)
   • ساختن فایل نهایی (build: مثلاً تبدیل TypeScript به JavaScript)
     ↓
۵. اگر همه تست‌ها پاس شد → برو به مرحله CD
     ↓
۶. مرحله CD شروع می‌شه:
   • کد رو به سرور اصلی منتقل کن (مثلاً AWS, DigitalOcean)
   • وابستگی‌ها رو اونجا هم نصب کن
   • دیتابیس رو بروزرسانی کن (Migration)
   • سرویس رو ری‌استارت کن
     ↓
۷. یه پیام توی تلگرام یا ایمیل میاد: "✅ استقرار با موفقیت انجام شد"
     ↓
۸. کاربرهای سایت الان کد جدید رو می‌بینن!
```

**کل این فرایند معمولاً کمتر از ۵ دقیقه طول می‌کشه!**

---

## چرا CI/CD اینقدر مهمه؟ (مزایا با مثال واقعی)

### ۱. **دیگه "روی ماشین من کار می‌کرد!" نداریم**
**قبل از CI/CD:**  
تو کد رو فرستادی به سرور، خراب شد. می‌گی: "چی؟! روی ماشین خودم که درست کار می‌کرد!"

**بعد از CI/CD:**  
سیستم خودکار کدت رو توی یه محیط تمیز (مثل یه کامپیوتر خالی) تست می‌کنه. اگه اونجا کار نکرد، بهت می‌گه "نه بابا، کدت مشکل داره!"

### ۲. **اشتباهات انسانی از بین می‌ره**
**قبلش:**  
ساعت ۲ شب، نصف خواب، توی سرور دستور `rm -rf /` زدی و کل سایت رو پاک کردی! 😅

**بعدش:**  
ربات همه کارها رو انجام میده، تو فقط کد رو پوش می‌کنی.

### ۳. **سرعت تیم بالا می‌ره**
**قبلش:**  
- ۱ ساعت کد زدن  
- ۳۰ دقیقه تست دستی  
- ۱ ساعت منتظر موندن برای دپلوی (چون فقط یه نفر دسترسی داره)  
- ۲۰ دقیقه آپدیت سرور

**بعدش:**  
- ۱ ساعت کد زدن  
- git push  
- ۵ دقیقه بعد، کدت روی سایت رفته!

### ۴. **بازگشت به نسخه قبل آسون می‌شه**
کد جدید خراب کرد سایت؟ با یه کلیک (یا حتی خودکار) برمی‌گردی به آخرین نسخه سالم.

## ابزارهای محبوب CI/CD (با توضیح ساده)

| ابزار | نوع | برای کی خوبه | مثال فایل تنظیمات |
|-------|------|--------------|-------------------|
| **GitHub Actions** | رایگان در گیت‌هاب | پروژه‌های کوچک و متوسط | `.github/workflows/main.yml` |
| **GitLab CI** | رایگان در گیت‌لب | تیم‌های حرفه‌ای | `.gitlab-ci.yml` |
| **Jenkins** | متن‌باز، سنگین | پروژه‌های خیلی بزرگ سازمانی | `Jenkinsfile` |
| **CircleCI** | ابری، سریع | استارتاپ‌ها | `.circleci/config.yml` |

## مثال عملی با GitHub Actions (برای یه پروژه Node.js)

فرض کن می‌خوای هر بار که کدت به `main` رسید، خودکار تست بشه و بره روی سرور.

**فایل `.github/workflows/deploy.yml` رو می‌سازی:**

```yaml
name: CI/CD Pipeline  # اسم پروفایل

on:
  push:
    branches: [ main ]  # فقط وقتی کد به main می‌ره

jobs:
  test:  # مرحله اول: CI (تست کردن)
    runs-on: ubuntu-latest
    steps:
      - name: گرفتن کد از گیت‌هاب
        uses: actions/checkout@v3
      
      - name: نصب Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      
      - name: نصب کتابخانه‌ها
        run: npm install
      
      - name: اجرای تست‌ها
        run: npm test
      
      - name: چک کردن کیفیت کد
        run: npm run lint

  deploy:  # مرحله دوم: CD (رفتن روی سرور)
    needs: test  # فقط اگه تست پاس شد
    runs-on: ubuntu-latest
    steps:
      - name: انتقال به سرور از طریق SSH
        uses: appleboy/scp-action@v0.1.4
        with:
          host: ${{ secrets.SERVER_IP }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          source: "./"
          target: "/var/www/myapp"
      
      - name: ری‌استارت کردن اپلیکیشن
        uses: appleboy/ssh-action@v0.1.5
        with:
          host: ${{ secrets.SERVER_IP }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /var/www/myapp
            npm install
            pm2 restart myapp
```

**توضیح ریز خط‌ها:**  
- `on.push.branches: [main]` → هر بار کد به main بره، این پروفایل اجرا بشه  
- `jobs.test` → اول همه چی رو تست کن  
- `jobs.deploy` → اگه تست پاس شد، برو روی سرور نصب کن  
- `secrets.XXX` → اطلاعات حساس رو توی تنظیمات گیت‌هاب ذخیره می‌کنی (رمزهای سرور)

## چالش‌های رایج در CI/CD (و راه حل ساده)

### ❌ **تست‌ها توی CI خراب می‌شه، ولی روی ماشین من درسته!**

**دلیل:** شاید روی ماشینت یه کتابخانه قدیمی داری، یا یه متغیر محیطی رو فراموش کردی.

**راه حل:** از Docker استفاده کن که محیط تست دقیقاً مثل سرور باشه.

### ❌ **پایپلاین خیلی طولانی می‌شه (۲۰ دقیقه!)**

**دلیل:** شاید داری تست‌های خیلی سنگین اجرا می‌کنی.

**راه حل:**  
- تست‌ها رو موازی کن (Parallel)  
- فقط بخش‌هایی که تغییر کردن رو تست کن (Cache کردن)  
- از GitHub Actions بزرگتر بخر (مثلاً ۸ هسته به جای ۲ هسته)

### ❌ **هر بار کل کتابخانه‌ها نصب می‌شه (اتلاف وقت)**

**راه حل:** از Cache استفاده کن:

```yaml
- name: Cache کردن node_modules
  uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('package-lock.json') }}
```

## خلاصه نهایی برای به خاطر سپردن

```
CI/CD = تست خودکار + استقرار خودکار

CI می‌گوید: "کدت خراب نیست، می‌شه با بقیه قاطیش کرد"
CD می‌گوید: "خُب حالا بذارش روی سرور که همه ببیننش"

قانون طلایی: 
هیچ کدی بدون CI به main نرسه!
هیچ mainی بدون CD به سرور نره!

نتیجه نهایی: 
شما فقط کد می‌زنی، بقیه کارها رو ربات انجام میده.
```

## مثال واقعی از زبون ساده (برای یادآوری)

**قبل از CI/CD:**  
من کد می‌زنم → ایمیل می‌زنم به تیم "بچه‌ها کد آماده" → یه نفر دستی میاره رو سرور تست → می‌گه "خرابه" → من می‌گم "روی من درسته" → دعوا می‌شه → ۳ ساعت معطلی → آخرش می‌فهمیم یه فایل رو فراموش کردم کامیت کنم 😅

**بعد از CI/CD:**  
من `git push` می‌کنم → ۲ دقیقه بعد پیام میاد "✅ تست‌ها پاس شدن، کدت رفت روی سرور زنده!" → من می‌رم چای می‌خورم 😎
