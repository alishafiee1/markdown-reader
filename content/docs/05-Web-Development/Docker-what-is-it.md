# 🐳 آموزش داکر (Docker) به زبان خودمونی و مثال‌های ملموس

## مقدمه: اصلاً چرا داکر به درد می‌خوره؟

فرض کن یه برنامه نوشتی (مثلاً با پایتون). روی لپ‌تاپ خودت همه چی درسته. اما می‌دی به دوستت، می‌گه «روی من کار نمی‌کنه! نسخه پایتون فرق داره! کتابخونه فلان نیست!». دو روز وقت می‌ذاری باهاش برای نصب کردن.

داکر میاد می‌گه: «برنامه رو به همراه همه چیزایی که نیاز داره (خود پایتون، کتابخونه‌ها، تنظیمات) بگذار توی یه جعبه. اسم این جعبه رو بگذار **کانتینر**. هر جا بری، همین جعبه رو بیار و اجرا کن. دیگر مهم نیست سیستم مقصد چه چیزایی نصب داره یا نداره.»

> مثال رستورانی که گفتم: داکر مثل اینه که غذای مورد علاقت رو به همراه قاشق و چنگال و دستمال و سس مخصوص، بگذاری توی یه **ظرف یکبار مصرف درب‌دار**. ببری هر خونه‌ای، بدون نگرانی بخوری. ظرف هم بعد از مهمونی دور انداخته بشه یا نه، اختیاری.

---

## یک مثال واقعی و ملموس

فرض کن یه وب‌اپ ساده با پایتون و Flask داری. برای اجرا نیاز داری:
- پایتون ۳.۱۰
- کتابخونه Flask
- (اختیاری) Redis

بدون داکر: باید روی هر ماشینی که می‌خوای برنامه رو اجرا کنی، اینا رو نصب کنی.

با داکر: یه فایل به اسم `Dockerfile` می‌نویسی (دستورالعمل):

```dockerfile
FROM python:3.10-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
```

بعد با دو تا دستور:

```bash
docker build -t myapp .
docker run -p 5000:5000 myapp
```

و برنامه بدون دردسر روی هر کامپیوتری که داکر داشته باشه اجرا می‌شه.

---

## اصل مطلب: «بسته» (Image) و «کانتینر» (Container)

خیلی مهمه که فرق این دو رو بلد باشی:

| اصطلاح | معادل فارسی پیشنهادی | توضیح خودمونی |
|--------|----------------------|----------------|
| **Image** | **بسته** | یه فایل فقط‌خواندنی که شامل برنامه و همه وابستگی‌هاست. مثل یه **دی‌وی‌دی نصب** ویندوز. |
| **Container** | **کانتینر** | وقتی از روی اون بسته یه نمونه اجرا می‌کنی، بهش می‌گن کانتینر. مثل **ویندوز نصب شده** روی یه کامپیوتر. |

از یک بسته می‌تونی چندین کانتینر مجزا اجرا کنی. بسته رو با `docker build` می‌سازی، کانتینر رو با `docker run` راه می‌ندازی.

---

## دستورات پرکاربرد داکر (جدول سریع)

این دستورات رو توی ترمینال (CMD، PowerShell، bash) می‌نویسی:

| دستور | چه می‌کنه |
|-------|-----------|
| `docker pull nginx` | بسته رو از مخزن دانلود می‌کنه (مثل اینستاگرام عکس گرفتن) |
| `docker images` | لیست بسته‌هایی که داری توی سیستمت |
| `docker build -t myapp .` | از روی Dockerfile یه بسته می‌سازه |
| `docker run myapp` | از روی بسته یه کانتینر می‌سازه و اجراش می‌کنه |
| `docker run -d -p 8080:80 nginx` | اجرا در پس‌زمینه (detached) و پورت ۸۰ کانتینر رو به پورت ۸۰۸۰ ماشین خودت متصل می‌کنه |
| `docker ps` | لیست کانتینرهای در حال اجرا |
| `docker ps -a` | همه کانتینرها (حتی متوقف شده) |
| `docker stop <id>` | کانتینر رو خاموش می‌کنه |
| `docker start <id>` | کانتینر خاموش رو روشن می‌کنه |
| `docker rm <id>` | کانتینر رو پاک می‌کنه (قبلاً باید stop شده باشه) |
| `docker rmi <image>` | بسته رو پاک می‌کنه |
| `docker exec -it <id> bash` | می‌ره توی کانتینر در حال اجرا و یه شل (ترمینال) باز می‌کنه |
| `docker logs <id>` | لاگ‌های کانتینر رو نشون می‌ده |
| `docker system prune -a` | **حواس‌ت باشد**: همه چیزای استفاده نشده رو پاک می‌کنه (مثل یه نظافت عمومی) |

## داکر کَمپوز (Docker Compose) برای چند کانتینر

وقتی برنامه‌ات هم وب‌سرویس داره هم دیتابیس، بجای اینکه هر کدوم رو جداگانه با `docker run` راه بندازی، یه فایل `docker-compose.yml` می‌نویسی و با `docker-compose up -d` همه رو یکجا بالا میاری.

---

## چالش‌های استفاده از داکر (که باید بلد باشی)

1. **یادگیری اولیه** درد داره، ولی بعد از چند بار تمرین عادی می‌شه.
2. **داده‌های دائمی**: کانتینرها ذاتاً زودگذرن. برای اینکه داده‌هات (مثل دیتابیس) موقع پاک شدن کانتینر از بین نره، باید از **ولوم** استفاده کنی.
3. **عملکرد روی ویندوز و مک** کمی کندتر از لینوکسه (چون داکر پشت صحنه یه لینوکس مجازی راه می‌ندازه).
4. **امنیت**: دسترسی به داکر یعنی دسترسی به ریشه سیستم. باید دقت کنی چه کسانی می‌تونن دستورات داکر رو اجرا کنن.

---

## 💾 ولوم (Volume) چی هست و به چه دردی می‌خوره؟

ولوم یعنی یه پوشه روی سیستم میزبان (هاست) که داکر اون رو مدیریت می‌کنه و می‌تونه به کانتینر وصل بشه. داده‌هایی که تو ولوم ذخیره می‌شن، حتی اگر کانتینر رو حذف کنی، باقی می‌مونن.

**کاربرد اصلی:** دیتابیس، آپلود فایل‌ها، لاگ‌ها، هر چیزی که می‌خوای بمونه.

**مثال:**

```bash
docker volume create mydb_data
docker run -v mydb_data:/var/lib/mysql mysql
```

حالا هر دفعه کانتینر رو عوض کنی، ولوم رو دوباره وصل می‌کنی و داده‌هات کوک هستن.

سه نوع روش ذخیره‌سازی داریم:

| نوع | توضیح خودمونی |
|-----|----------------|
| **Volume** | مدیریت شده توسط داکر، بی‌خیال مسیرش. بهترین گزینه برای داده دائمی. |
| **Bind mount** | یه پوشه از روی کامپیوتر خودت رو می‌چسبونی به کانتینر. خوب برای توسعه که کد رو عوض می‌کنی فوری تو کانتینر ببینی. |
| **tmpfs** | توی رم ذخیره می‌شه، با حذف کانتینر از بین می‌ره. مناسب داده‌های حساس موقتی. |

---

## 🔐 دسترسی‌ها و محدودیت‌ها در داکر

داکر از امکانات کرنل لینوکس استفاده می‌کنه تا کانتینرها رو از هم و از سیستم میزبان جدا کنه:

- **Namespace**: هر کانتینر فکر می‌کنه تنهاست (فرآیندها، شبکه، کاربران خودش رو داره).
- **Cgroup**: می‌تونی محدودیت CPU و رم برای هر کانتینر تعیین کنی (مثلاً بگی این کانتینر بیشتر از ۵۰٪ CPU استفاده نکنه).
- **Linux Capabilities**: به کانتینرها root نمی‌ده، بلکه یه سری مجوزهای محدود می‌ده.

**نکته امنیتی مهم**: هر کسی که بتونه دستورات داکر رو اجرا کنه، عملاً دسترسی root به کل سیستم داره. پس گروه `docker` رو با احتیاط مدیریت کن.

---

## 🔄 بروزرسانی برنامه در داکر بدون از دست رفتن داده‌ها

روش استاندارد:

1. **بسته جدید** بساز: `docker build -t myapp:v2 .`
2. کانتینر قدیمی رو متوقف و حذف کن: `docker stop myapp && docker rm myapp`
3. کانتینر جدید رو با **همون ولوم** راه بنداز: `docker run -v mydata:/data -p 80:80 myapp:v2`

برای اینکه لحظه‌ای قطعی نداشته باشی (مثل سایت‌های شلوغ) از روش **آبی-سبز** استفاده کن: دو کانتینر همزمان داریم، یکی فعلی (سبز)، یکی جدید (آبی). ترافیک رو یکی یکی جابه‌جا می‌کنیم.

---

## 📛 اسم‌گذاری بسته‌ها و مخازن خارجی

فرمت کلی اسم یک بسته:

```
[registry-host[:port]/][namespace/]repository[:tag]
```

- **registry-host** (مخزن): آدرس جایی که بسته‌ها نگهداری می‌شن. پیش‌فرض `docker.io` (داکرهاب).
- **namespace** (فضای نام): اسم کاربری یا سازمان.
- **repository** (مخزن): اسم پروژه.
- **tag** (برچسب): نسخه (مثلاً `v1`، `latest`).

**آیا مخزن می‌تونه جای دیگه باشه؟** بله، کاملاً. می‌تونی از `ghcr.io` (گیت‌هاب) استفاده کنی یا مخزن خودت رو با `docker run -d -p 5000:5000 registry` راه بندازی.

**اسم‌ها باید یونیک باشن؟** در سطح یک مخزن، ترکیب `namespace/repository:tag` باید یکتا باشه. اما دو مخزن مختلف می‌تونن بسته‌های هم‌نام داشته باشن.

## 🧱 نحوه دانلود بسته‌ها و لایه‌بندی

وقتی `docker pull nginx` می‌زنی، داکر همه لایه‌های بسته رو دانلود نمی‌کنه، فقط لایه‌هایی که قبلاً نداری رو میاره. هر `RUN`، `COPY`، `ADD` در Dockerfile یه لایه می‌سازه. اگر لایه‌ای تغییری نکرده باشه، از کش استفاده می‌شه.

پس ترتیب دستورات توی Dockerfile خیلی مهمه: **چیزایی که کم عوض می‌شن (نصب وابستگی‌ها) رو بذار اول، چیزایی که زیاد عوض می‌شن (کد برنامه) رو بذار آخر.**

---

## 📦 آیا وابستگی‌ها حجم بسته رو زیاد می‌کنن؟ (بله!)

هر کتابخونه‌ای که توی لایه‌ها نصب بشه، به حجم نهایی اضافه می‌شه. اما راه‌های کاهش حجم:

1. **ساخت چندمرحله‌ای (Multi-stage build)**:  
   یه مرحله برای بیلد (با کامپایلر و ابزار سنگین)، یه مرحله برای اجرا (فقط فایل نهایی). مثال:

```dockerfile
FROM golang:1.19 AS builder
WORKDIR /src
COPY . .
RUN go build -o myapp

FROM alpine:latest
COPY --from=builder /src/myapp /usr/bin/myapp
CMD ["myapp"]
```

2. **استفاده از بسته‌های پایه سبک** مثل `alpine`، `slim`.
3. **حذف کش‌های نصب‌کننده** در همان `RUN`:

```dockerfile
RUN apt-get update && apt-get install -y package && rm -rf /var/lib/apt/lists/*
```

---

## 🤔 آیا وابستگی‌ها رو بعد از استقرار می‌شه جدا دانلود کرد؟

به طور استاندارد **خیر**. وابستگی‌هایی که برنامه برای اجرا نیاز داره باید داخل بسته باشن. در غیر این صورت اصل «قابل حمل بودن» داکر نقض می‌شه.

**استثنا**: داده‌های حجیم مثل مدل‌های یادگیری ماشین یا ویدیوها رو توی ولوم می‌ریزی، نه داخل بسته. برنامه تو می‌تونه این داده‌ها رو زمان اجرا از یک جای دیگه (مثل S3) دانلود کنه و توی ولوم ذخیره کنه.

---

## 🧾 Dockerfile به زبان خودمونی

Dockerfile یه فایل متنی ساده‌ست که به داکر می‌گه چطور بسته رو بسازه. بیا با مثال آش رشته توضیح بدم:

```dockerfile
# FROM = از روی این بسته پایه شروع کن (مثل اینه که بگی «اول یه قابلمه بردار»)
FROM python:3.10-slim

# WORKDIR = تو بسته یه پوشه درست کن و برو تو اون (مثل میز کار)
WORKDIR /app

# COPY = فایل رو از روی کامپیوتر خودت بذار تو بسته (مثل چاقو انداختن تو ظرف مسافرتی)
COPY requirements.txt .

# RUN = موقع ساخت بسته این دستور رو اجرا کن (مثل بریز آب رو بذار بجوشه)
RUN pip install -r requirements.txt

# دوباره COPY برای کد اصلی
COPY . .

# CMD = وقتی کانتینر اجرا شد این دستور رو بزن (مثل بزن به خورد)
CMD ["python", "app.py"]
```

**نکات کلیدی Dockerfile**:
- هر `RUN`، `COPY`، `ADD` یه لایه می‌سازه. سعی کن تعداد لایه‌ها رو کم کنی (مثلاً با چسبوندن دستورات با `&&`).
- `CMD` فقط یک بار می‌تونه باشه. اگر چند تا بنویسی، فقط آخری اجرا می‌شه.
- `ENTRYPOINT` مشابه `CMD` ولی سخت‌تر قابل تغییره. معمولاً برای تعیین برنامه اصلی استفاده می‌شه.

---

## 🌐 کار با مخزن (Registry) مثل داکرهاب

داکرهاب مثل گیت‌هاب برای بسته‌های داکره. مراحل:

1. **ثبت‌نام** در [hub.docker.com](https://hub.docker.com).
2. **وارد شدن** از خط فرمان: `docker login`
3. **جستجو**: `docker search ubuntu`
4. **دانلود**: `docker pull ubuntu:22.04`
5. **آپلود بسته خودت**:  
   - بسته رو با فرمت `username/nama:tag` برچسب بزن: `docker tag myapp alireza/myapp:v1`  
   - آپلود کن: `docker push alireza/myapp:v1`

دیگران هم می‌تونن با `docker pull alireza/myapp:v1` بسته تو رو دانلود کنن.

---

## جمع‌بندی نهایی (برای اینکه یادت نره)

- **داکر** یعنی برنامه + وابستگی‌ها داخل یه کانتینر.
- **بسته (Image)** = قالب فقط خواندنی.
- **کانتینر (Container)** = نمونه اجرایی.
- **ولوم (Volume)** = کیف پول دائمی داده‌ها.
- **Dockerfile** = دستور العمل ساخت بسته.
- **داکرهاب** = اینستاگرام بسته‌ها.

با تمرین این دستورات توی ترمینال، ظرف یکی دو روز بهت می‌چسبه. اگر چیزی مبهم بود، برگرد به همون مثال «آش رشته». آشپز خارجی (داکر) فقط و فقط به دستورالعملی که بهش می‌دی (Dockerfile) عمل می‌کنه.

حالا برو یه `docker run hello-world` بزن و لبخند بزن. 😊
